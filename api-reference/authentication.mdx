title: "Authentication"
description: "Establish a wallet-bound session by signing a one-time challenge — no email, password, or API key."
---

The Aura API uses **wallet-bound sessions**, the same mechanism as the Aura
app (whose embedded wallet signs the challenge with its groupless secp256k1
key). There are no API keys.

- **Reads are public.** Every `GET` endpoint works without authentication
  (subject to the anonymous per-IP [rate limits](/api-reference/rate-limits)).
- **Writes require a session.** Mutating endpoints (comments, watchlist,
  notifications, profile writes) and the build-tx endpoints under `/v1/tx/*`
  require a valid session **cookie** plus a **CSRF header**.

The session authorises off-chain writes only. On-chain actions are returned
as *unsigned* transactions that your wallet signs locally — the API never
holds keys and the session can never move funds.

## Establishing a session

Two-step wallet-signature flow:

<Steps>
  <Step title="Get a one-time challenge">
    ```bash
    curl -X POST https://api.aurabets.io/v1/auth/challenge \
      -H 'content-type: application/json' \
      -d '{"address": "3Cef...your-wallet"}'
    ```

    Response:

    ```json
    {
      "challenge": "Sign this message...",
      "expiresAt": "2026-07-01T19:00:00Z"
    }
    ```

    `address` is optional — omit it when the wallet address is only known
    after authentication, then pass the exact `challenge` text in the next
    step.
  </Step>
  <Step title="Sign with your wallet">
    Use your wallet's `signMessage` helper with `hasher: "alephium"` to sign
    the `challenge` string. `keyType` identifies the signature scheme:

    | `keyType` | Wallet |
    | --- | --- |
    | `gl-secp256k1` | Groupless secp256k1 — what the Aura app's embedded wallet uses (addresses start with `3`) |
    | `default` | Grouped secp256k1 — standard Alephium wallets and tooling |
    | `gl-webauthn` | Legacy passkey (WebAuthn) signatures — still verified for older signers |
  </Step>
  <Step title="Exchange the signature for a session">
    ```bash
    curl -X POST https://api.aurabets.io/v1/auth/session \
      -H 'content-type: application/json' \
      -c cookies.txt \
      -d '{
        "address":   "3Cef...your-wallet",
        "publicKey": "0233...your-pubkey",
        "signature": "abcd...sig",
        "keyType":   "gl-secp256k1"
      }'
    ```

    Response:

    ```json
    {
      "address": "3Cef...your-wallet",
      "csrf": "5f2a...token",
      "created": false
    }
    ```

    The session itself is set as an **HttpOnly cookie** (30-day TTL with
    sliding refresh). Keep the `csrf` value — every mutating request must
    echo it back in the `x-csrf-token` header.

    First sign-in also creates the account server-side with a **generated
    unique username**; manage it via `GET/PATCH /v1/users/me` and
    `GET /v1/users/username-available`.
  </Step>
</Steps>

## Making authenticated requests

Send the session cookie **and** the CSRF header on every mutating request:

<CodeGroup>
```bash curl
curl -X POST https://api.aurabets.io/v1/watchlist \
  -b cookies.txt \
  -H "x-csrf-token: $CSRF" \
  -H 'content-type: application/json' \
  -d '{"marketId": 42}'
```

```ts TypeScript (fetch)
const res = await fetch('https://api.aurabets.io/v1/watchlist', {
  method: 'POST',
  credentials: 'include', // sends the HttpOnly session cookie
  headers: {
    'content-type': 'application/json',
    'x-csrf-token': csrf,
  },
  body: JSON.stringify({ marketId: 42 }),
})
```

```py Python (requests)
import requests

s = requests.Session()
# ...challenge + session calls populate s.cookies...
res = s.post(
    'https://api.aurabets.io/v1/watchlist',
    headers={'x-csrf-token': csrf},
    json={'marketId': 42},
)
```
</CodeGroup>

`GET` requests never need the CSRF header — only the cookie, and only on
endpoints that return caller-specific data.

## Session lifecycle

| Endpoint | Purpose |
| --- | --- |
| `POST /v1/auth/challenge` | Mint a one-time nonce to sign (5-minute expiry). |
| `POST /v1/auth/session` | Verify the signature, create/refresh the account, set the session cookie. Sign-in **and** registration. |
| `GET /v1/auth/session` | Check whether the cookie is still valid and recover the CSRF token. |
| `POST /v1/auth/logout` | Clear the session + CSRF cookies. |

<Warning>
  Build-tx endpoints under `/v1/tx/*` additionally enforce that the
  transaction's **signer equals the session wallet** — a session for wallet A
  can never build a transaction that spends from wallet B.
</Warning>
