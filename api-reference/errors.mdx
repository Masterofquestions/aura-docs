---
title: "Errors"
description: "Standard error envelope and HTTP status codes used across the API."
---

The Aura API returns every error as JSON with a stable envelope:

```json
{
  "error": {
    "code": "invalid_price",
    "message": "price must be in (0, 1000); got 1000",
    "details": { "field": "price" }
  }
}
```

| Field | Description |
| --- | --- |
| `error.code` | Stable, machine-readable error code in `snake_case`. Don't switch on `message`. |
| `error.message` | Human-readable error description. |
| `error.details` | Optional structured context (present on validation and rate-limit errors). |

The HTTP status code is carried on the response itself, not duplicated in
the body.

## Status codes

| Status | When |
| --- | --- |
| `400 Bad Request` | Validation failed — schema, range, or business rule (`details` carries the zod issues for schema failures). |
| `401 Unauthorized` | No valid session cookie. Sign in via `/v1/auth`. |
| `403 Forbidden` | Session valid but not allowed — missing/invalid `x-csrf-token`, or the tx signer doesn't match the session wallet. |
| `404 Not Found` | The resource doesn't exist (or you don't own it, for wallet-scoped reads). |
| `409 Conflict` | Idempotency / state conflict (e.g. a sponsorship `txId` recorded with different parameters). |
| `429 Too Many Requests` | Rate limit exceeded. See [Rate Limits](/api-reference/rate-limits). |
| `500 Internal Server Error` | Bug on our side — the original error is logged server-side, never leaked in the response. |
| `502 Bad Gateway` | The Alephium node refused to build a tx, or an upstream (node / indexer) is unhealthy. Retry. |
| `503 Service Unavailable` | Database or indexer temporarily unavailable. Retry with backoff. |

## Common error codes

| Code | Meaning |
| --- | --- |
| `bad_request` | Request body or query failed schema validation. |
| `unauthorized` | Session required or invalid (sign in via `/v1/auth`). |
| `csrf_failed` | Missing or invalid `x-csrf-token` header on a mutating request. |
| `signer_mismatch` | The session is bound to a different wallet than `signerAddress`. |
| `rate_limited` | Per-IP or per-wallet rate limit exceeded (`details` carries `max` + `ttl`). |
| `not_found` / `market_not_found` / `draft_not_found` | Resource doesn't exist. |
| `invalid_quantity` / `invalid_price` / `invalid_outcome` | Order parameters violate the contract's rules (lot size, price tick range, yes/no). |
| `field_too_long` | A proposal field exceeds its UTF-8 byte limit (limits mirror the on-chain checks). |
| `bond_mismatch` | A caller-supplied `bondAmount` doesn't match the on-chain bond. Omit it to use the on-chain value. |
| `build_failed` | The Alephium node refused to build the unsigned tx (returned as `502`). |
| `db_error` | Database read/write failed (returned as `503`). |
| `internal_error` | Unhandled server error (returned as `500`). |

## Idempotency & retries

- All `GET` endpoints are safe to retry.
- `POST /v1/tx/*` endpoints build (never submit) transactions — retrying is
  always safe. Note the returned `unsignedTx` may differ between calls as
  the wallet's UTXO set changes; only the transaction your wallet actually
  signs and submits matters.
- `POST /v1/rewards/sponsorships` is idempotent on `txId` — repeat calls
  with the same `txId` return the existing row.
